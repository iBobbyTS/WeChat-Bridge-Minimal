# WeChat Bridge Minimal 实现计划

## 摘要
- 从空仓库重建一个最小 TypeScript/Node 版本，不复制 `/Users/ibobby/Documents/WeChat-Bridge` 或 AGPL 参考项目代码，只参考协议和行为边界。
- 功能只包含：微信 QR 登录、后台长轮询、单 Codex CLI thread 对话、本地 send-only HTTP API、macOS launchd 开机自启。
- 微信协议边界参考官方 Tencent/openclaw-weixin；CLI 产品形态参考 UNLINEARITY/CLI-WeChat-Bridge，但不引入其实现代码。

## 关键实现
- 项目骨架：`package.json`、`tsconfig.json`、`src/`、`test/`，使用 ESM TypeScript、Node >=24、`node:test`、`tsx`、`qrcode-terminal`。
- 微信层：实现官方 iLink 薄客户端，支持 QR 登录、状态轮询、凭证保存、`getupdates`、`sendmessage`；状态目录默认 `~/.wechat-bridge-minimal`，凭证/token 文件 `chmod 600`。
- 对话层：所有微信入站文本进入同一个串行队列；首条消息运行 `codex exec` 创建会话并从 `~/.codex/sessions` 记录 thread id，后续消息固定用 `codex exec resume <threadId>`；自动上下文压缩交给 Codex CLI 原生会话机制，bridge 不维护自定义长上下文。
- 安全默认：只接受登录返回的目标用户消息；默认 Codex 参数为 `--sandbox read-only`、`--ask-for-approval never`，可通过环境变量覆盖；不支持 `/` 命令、群聊、媒体、审批转发或多线程路由。
- 本地 API：`serve` 启动 HTTP API，`POST /send` 接收指定 IP + bearer token 请求，把 token 对应名称和请求消息合并为 `name:\nmessage` 发到微信；`GET /health` 返回健康状态。token 文件每次请求重新读取，支持热更新。
- Token 管理：CLI 提供 `send-api-token ensure-defaults|list|add|remove`；默认可创建 `MacBook Pro`、`PC-4070`、`PC-4060` 三个随机 token，但必须显式执行命令生成。
- macOS 自启：提供 launchd 安装/状态/重启/日志脚本，生成 `~/Library/LaunchAgents/com.ibobby.wechat-bridge-minimal.plist`，环境文件放在 `~/.config/wechat-bridge-minimal/service.env`，日志放在 `~/.wechat-bridge-minimal/logs/`。

## 对外接口
- CLI：
  - `npm run build`
  - `npm run test`
  - `npm run weixin:login`
  - `npm run weixin:serve`
  - `npm run weixin:send-api-token -- ensure-defaults|list|add|remove`
  - `npm run service:install|status|restart|logs`
- HTTP：
  - `GET /health` -> `{ "success": true }`
  - `POST /send`，header `Authorization: Bearer <token>`，body `{ "text": "..." }`
  - 错误码固定为 `forbidden_ip`、`unauthorized`、`invalid_json`、`request_body_too_large`、`text_required`、`wechat_context_required`、`send_failed`。
- 主要环境变量：
  - `WECHAT_BRIDGE_STATE_DIR`
  - `WECHAT_SEND_API_HOST`
  - `WECHAT_SEND_API_PORT`
  - `WECHAT_SEND_API_ALLOWED_IPS`
  - `WECHAT_TARGET_USER_ID`
  - `CODEX_BIN`
  - `CODEX_CWD`
  - `CODEX_MODEL`
  - `CODEX_PROFILE`
  - `CODEX_SANDBOX`
  - `CODEX_APPROVAL`

## 测试计划
- 单元测试覆盖微信 API 请求构造、QR 状态处理、账号/上下文 token 存储、入站文本解析、出站消息 payload。
- API 测试覆盖 IP 白名单、bearer token 校验、恒定时间比较、body 大小限制、错误码、token 热加载、`name:\nmessage` 合并格式。
- Codex 适配器测试用 fake spawn/session 文件验证首次创建 thread、后续 resume、串行队列、失败回退和输出提取。
- launchd 测试验证 plist 生成字段、环境文件默认值、日志路径。
- 完成后运行 `npm run typecheck`、`npm run test`、`npm run build`。

## 假设与维护性判断
- 假设首版只面向 macOS，本地 API 默认绑定 `127.0.0.1`，需要局域网访问时由环境变量显式改为 `0.0.0.0` 并配置允许 IP。
- 假设 send-only API 不进入 Codex 对话，只负责把外部机器消息转发到微信。
- 维护性判断：实现会拆成微信协议、状态存储、Codex runner、本地 API、service 脚本五个边界清晰的小模块；复杂度主要在登录和 Codex thread 持久化，需用 mock 测试保护，不建议引入旧项目的大型 coordinator。
